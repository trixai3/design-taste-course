设计判读 · The Design Read

动手之前先说的那一句话。是受众决定美学，而不是你的口味。 来源：design-taste-frontend §0, §2。

那句公式

把它读作：给 [某人群 audience] 的 [某种页面 page kind]，
用 [某种气质 vibe] 的语言，
倾向于 [某个基座 foundation]。

四个部分

1 · 页面类型 Page kind

落地页、作品集、服务页、编辑页、仪表盘。定下基本规则。

2 · 受众 Audience

谁会扫这个页、他们需要什么。这才是真正决定旋钮的一环。

3 · 气质 Vibe

两三个词：求稳、高级编辑感、俏皮、粗野、无障碍优先。

4 · 基座 Foundation

需求需要时就用真实的设计系统，否则用一个诚实的美学家族加一款有名字的字体。

需求 → 基座对照表

如果需求读起来像……就用为什么
企业级 SaaS、仪表盘、微软味Fluent UI官方微软 token，无障碍已就绪。
IBM 式 B2B 分析、数据密集Carbon成熟的数据密度范式。
谷歌味、Material 风格产品Material 3官方、可主题化。
GitHub 式开发者工具 / 社区PrimerGitHub 官方系统。
英国公共部门服务GOV.UK Frontend法规上被预期。
美国公共部门 / 求稳USWDS同上，用于美国政府。
现代 SaaS 且组件你自己拥有shadcn/ui代码归你；永远别直接发默认态。
独立 / 小团队 AI 营销Tailwind v4独立构建的默认选择。
代理、编辑、粗野、高端native CSS没有官方包。用一款有名字的字体去建，并诚实标注。

诚实原则：如果需求匹配到某个真实系统，就装官方包。不要手搓它的 CSS，也不要把某个美学趋势 （glassmorphism、bento）说成官方系统。
